# 我重构了知识库的文本预处理——发现「定语」比「主语宾语」重要得多

## 01 一个让我停下来重写的细节

知识库上线后，我遇到了一个很奇怪的问题：

搜「跨模态图像分割方法」能命中，但搜「跨模态」反而排到后面去了。

直觉告诉我问题出在文本切片和实体提取上。打开角色表一看：

```
实体: 「方法」 → 出现 47 次
　作为主语出现在: sphere_001, sphere_015, sphere_023 ...
　作为宾语出现在: sphere_032, sphere_047 ...
实体: 「模型」 → 出现 34 次
实体: 「性能」 → 出现 31 次
```

「方法」「模型」「系统」「性能」——这些词出现频率最高，但信息量几乎为零。

一个知识库里 200+ 条技术笔记，「方法」出现几十次，但它能告诉你什么？什么都不能。

真正能让两条笔记区隔开来的，是「跨模态图像分割」「基于深度学习的」「长距离依赖」这些定语。

## 02 主语和宾语不是语义角色，是句法角色

传统 NLP 的文本理解范式高度依赖主谓宾（SVO）结构。spaCy 提供依存句法分析，能标出：

```
[方法] NOUN  nsubj  →  「方法是…」
[性能] NOUN  dobj   →  「展现性能」
```

主语和宾语在语法分析里很重要，但放在知识库里，它们的价值被高估了。

原因很简单：**主语和宾语告诉你「这块在讲什么领域」，定语才告诉你「这块具体是哪一条」。**

拿一个典型的技术句子举例：

> 基于深度学习的跨模态图像分割方法在医疗影像分析中展现出优异性能。

传统 SVO 抽取：
- 主语：「方法」← 200 条里 47 条都有它
- 宾语：「性能」← 更泛

定中短语抽取（我做的）：
- **跨模态图像分割方法** ← 唯一标识
- **医疗影像分析** ← 另一个唯一标识
- **基于深度学习的跨模态图像分割方法** ← 完整上下文

你想，一个搜索「时序预测模型」的人，真正匹配的是「模型」还是「时序预测」？

当然是后者。定语空间的密度比中心语高一个数量级。

## 03 怎么用 spaCy + 规则做定中短语提取

核心思路很简单：**不要依赖主语/宾语，而是找定语-中心语关系。**

我用了三层策略，逐级互补：

### 第一层：spaCy 依存句法（覆盖短修饰）

spaCy 中文模型用 `compound:nn` 标记名词复合（「Transformer架构」）、`amod` 标记形容词修饰（「优异性能」）、`nmod:assmod` 标记领属修饰（「平台的底层」）。

这层跑得最快，一次遍历 O(n) 完成。

### 第二层：「的」锚定规则（覆盖长定语）

中文里「的」字结构几乎无歧义。「X的Y」中的 X 就是定语，Y 就是中心语。这条规则精度极高，覆盖了 spaCy 搞不定的长定语如「基于深度学习的跨模态图像分割方法」。

### 第三层：复合名词解构（覆盖无「的」术语）

以常见名词后缀（法、模型、系统、算法、架构……）结尾的无「的」复合名词，如「消息队列」「特征空间度量学习方法」。用后缀匹配解构出定语和中心语。

三层互补后，同样的 9 个测试句：

```
「Transformer架构」——旧: ❌  新: ✅ (compound:nn)
「标准测试集」   ——旧: ❌  新: ✅ (compound:nn)
「平台」+「的」+「底层」——旧: ❌  新: ✅ (nmod:assmod)
「基于深度学习的跨模态图像分割方法」——旧: ✅ (的锚定)  新: 保持
```

## 04 角色表的重构——从「主语/宾语」到「短语/中心语/定语」

这是最核心的改动。旧的角色表是：

```python
# 旧角色表
EntityInfo.as_subject: [...球体ID...]    # 方法作为主语
EntityInfo.as_object:  [...球体ID...]    # 方法作为宾语
```

新的角色表是：

```python
# 新角色表
EntityInfo.as_phrase:  [...球体ID...]    # 完整定中短语出现
EntityInfo.as_head:    [...球体ID...]    # 中心语裸名词出现
EntityInfo.as_attributive: [...球体ID...] # 定语单独出现
```

检索桥接逻辑也相应改变：

- **shared_phrase（最强）**：两个球体共享相同的定中短语
- **shared_head（中等）**：两个球体中心语相同（如都讨论「方法」）
- **shared_attributive（较弱）**：两个球体共享定语（如都有「跨模态」）

## 05 文档级术语：给每个切片贴上「全文档标签」

另一个改进是：切片之后，每个球体不再只有切片本身的信息，还会带上整个文档的关键术语。

```
文档「重力知识库架构分析.txt」
  ├─ AttrHeadExtractor → 提取 32 个关键术语
  │  (跨模态图像分割方法、医疗影像分析、自注意力机制……)
  │
  ├─ chunker → 切片 1（正文含「该方法」）
  │     └─ doc_terms: [跨模态图像分割方法, 医疗影像分析, ...]
  │
  └─ chunker → 切片 2（正文含「该模型」）
        └─ doc_terms: [跨模态图像分割方法, 医疗影像分析, ...]
```

这样即使切片正文只写「该方法」，检索时也能通过元数据里的「跨模态图像分割方法」命中。解决了代词指代的信息丢失问题。

## 06 一些数据

上传一篇 230 字的技术笔记片段，实际用时：

| 阶段 | 耗时 |
|------|------|
| 文档级术语抽取 | 0.87s |
| 切片 + 嵌入 | 2.3s |
| 角色表构建 | 4.2s |
| 聚类 | 8.8s |
| 总计 | ~17s |

抽到的 32 个术语覆盖了文档的全部核心概念，包括 spaCy 的 `compound:nn` 和规则的「的」锚定。spaCy sm 小模型的精度有限（46MB 的模型），部分长复合词如「长距离依赖关系」因为被解析为动宾结构而漏过——这需要更大模型或更复杂的组合检测来弥补，留待下次。

---

这次改动的核心收获其实是三句话：

1. **主语宾语是句法角色，定语才是鉴别器**
2. **规则和深度学习不是替代关系，是互补关系**
3. **切片丢失的信息，可以用文档级元数据补回来**
