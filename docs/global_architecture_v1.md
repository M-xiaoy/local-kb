# 重力知识库·系统架构 v1

> 全局视角：从原始文本输入到最终答案输出的完整链路。
> 每个组件不是孤立功能，是信息流中的一个节点——耦合越紧，信息损耗越少。

---

## 一、系统全景

```
                         ┌──────────────────────────────────────┐
                         │          反馈学习环 (Phase 3)        │
                         │  用户交互 → 隐式反馈 → 球体质心调整  │
                         └──────────────────────────────────────┘
                                      ▲
                                      │ 反馈信号
                                      ▼
┌──────────┐   ┌────────────┐   ┌────────────┐   ┌────────────┐   ┌──────────┐
│          │   │            │   │            │   │            │   │          │
│  输入层   │──▶│  精炼层    │──▶│  存储层    │──▶│  检索层    │──▶│  生成层   │
│          │   │            │   │            │   │            │   │          │
│ Raw Text │   │ Clean Text │   │ Spheres    │   │ Query      │   │ Answer   │
```

## 二、逐层设计

### Layer 0: 输入层 — 多源接入

**职责：** 把各种来源的原始文本统一成标准格式。

```
输入源:
  ├── Markdown 文件（主力，技术笔记）
  ├── PDF（论文/文档，需 OCR 或解析）
  ├── 网页（知乎文章/博客，需 HTML 清洗）
  ├── 对话记录（溪流数据，需结构化）
  └── API 输入（实时抓取的内容）

标准化输出:
  {
    "source": "github/local/file",
    "raw_text": "...",
    "meta": {
      "title": "...",
      "date": "2026-07-16",
      "tags": ["AI", "RAG"],
      "source_type": "technical_note"
    }
  }
```

**与精炼层的耦合：** 输入层不处理文本，只负责把不同来源的原始内容归为统一格式。精炼层只认 `raw_text`，不关心来源。

---

### Layer 1–2: 精炼层 — 从文本到结构化知识

**职责：** 把原始文本变成「带主语/宾语/关系的结构化片段」。

```
Layer 1: 文本清洗 + 结构分析
  输入: raw_text
  输出: 干净的句子序列 + 代词标记 + 引号内容

Layer 2: 候选提取 + 语义选择 + 片段注入
  输入: 句子序列 + 代词标记
  输出: 结构化片段列表

结构化片段:
{
  "fragment_id": "doc_001_p3_s2",
  "text": "经过实验验证，效果优于传统RAG",
  "explicit_subject": "重力知识库",
  "explicit_object": "传统RAG",
  "relations": [
    {"predicate": "优于", "subject": "重力知识库", "object": "传统RAG"}
  ],
  "quality_score": 0.85
}
```

**与存储层的耦合：** 精炼层的输出直接决定存储层能学到什么关系。主语噪音大 → 角色表不准；宾语噪音大 → 层级生长出错。**精炼层是全局质量的瓶颈。**

---

### Layer 3–4: 存储层 — 球体 + 向量 + 层级 + 角色

**职责：** 管理四种知识表示形态，保持它们之间的双向一致性。

```
四种知识形态：
                              ┌──────────────┐
                              │   概念球体     │  ← 语义空间中的概念锚点
                              │  SphereStore  │
                              └──────┬───────┘
                                     │ 每个球体对应
                            ┌────────┼────────┐
                            ▼        ▼        ▼
                     ┌──────────┐ ┌──────┐ ┌──────────┐
                     │  FAISS   │ │ 层级  │ │ 角色表    │
                     │  向量索引 │ │ 父/子 │ │ 共现关系   │
                     └──────────┘ └──────┘ └──────────┘

存储层内部耦合关系:

1. 概念球体 ↔ FAISS 向量
   每个球体有一个 embedding（来自 embedding_source：title/description/text）
   向量更新 → 球体同步（不是懒惰同步，是写入时原子更新）

2. 概念球体 ↔ 层级
   每个球体有 level（0=无层级, 1=一级, 2=二级, ...）
   层级生长 → 更新球体的 level/parent_id/child_ids
   反方向：球体分裂或合并 → 触发层级重建（增量而非重置）

3. 概念球体 ↔ 角色表
   角色表存储 (source, predicate, target) 三元组
   source/target 指向球体的 concept_id
   角色表更新 → 球体的 level 可能提升（出现频次高的从三级升为一级）

4. FAISS ↔ 角色表 ↔ 层级（检索时的三方联动）
   FAISS 检索返回候选球体集合
   角色表扩展：候选球体作为主语 → 扩展出其宾语角色
   层级扩展：一级球体 → 扩展出其二级子球体
   这是检索层的「三方联动」基础
```

**关键设计：四种知识形态不各自独立，它们共享同一个 concept_id 索引。删除/更新一个球体时，FAISS / 层级 / 角色表同时得到通知。**

---

### Layer 5–6: 检索层 — 多路召回 → 多样性排序 → 场域加权

**职责：** 把用户查询变成最相关的文档片段列表。

```
输入: 用户查询 (query text)
输出: 排序后的 fragment 列表

检索管道:

Step 1: 语义检索（FAISS）
  行为: query → embedding → FAISS 搜索 top-K 球体
  输出: [(concept_id, score), ...]

Step 2: 角色展开（RoleExpander）
  行为: 对每个候选球体，查角色表
        如果是主体 → 扩展出与其共现的客体
        如果是客体 → 扩展出与其共现的主体
  输出: [(concept_id, score, expanded_relations), ...]

Step 3: 层级展开（HierarchyExpander）
  行为: 对 Step 1~2 中 level=1 的球体
        查其 child_ids，递归展开（最多深度 2）
  输出: [(concept_id, score, level, parent_id), ...]

Step 4: 质量场加权（FieldWeighting）
  行为: query 的关键词 TF → 与球体的 term_weights 匹配
  score = 0.7 × semantic_score + 0.3 × field_score
  输出: 重新排序的候选列表

Step 5: 多样性排序（DiversitySorter）
  行为: MMR 算法，λ 控制多样性和相关性的平衡
        惩罚：来自同一个簇的多个候选得分过高
  输出: 最终排序的候选列表

Step 6: 簇级层次索引（ClusterRouter）
  行为: 先找 top-2 簇，再在簇内搜索
        如果 top-5 候选 ≥3 个来自同一个簇
        → 查询向量向该簇质心漂移 15%（引力漂移）
  输出: 调整后的候选列表
```

**与生成层的耦合：** 检索层输出候选列表 + 每个候选的结构化片段（含主语/宾语/关系）。生成层不需要重新理解文本，直接使用结构化信息。

---

### Layer 7: 生成层 — 答案生成

**职责：** 把检索结果变成自然语言答案。

```
输入: 检索层输出的候选片段列表 + 用户查询
输出: 自然语言答案

生成策略:

Ollama 本地模式:
  输入 prompt = 
    "基于以下知识片段回答用户问题：\n"
    + "\n---\n".join(fragments)
    + "\n用户问题: " + query

DeepSeek API 模式:
  同上，但切换后端

Agent 模式（未来）:
  多轮对话，每轮检索 → 生成 → 验证 → 补充
```

**与检索层的耦合：** 生成层只消费检索层的输出，不直接访问存储层。检索层的排序质量直接决定生成层的输入质量。

## 三、核心组件之间的双向耦合

```
精炼层 ────────→ 存储层: 结构化片段 → 更新球体/向量/角色表/层级
存储层 ────────→ 检索层: 提供多形态的知识表示
检索层 ────────→ 生成层: 候选片段列表
生成层 ────────→ 精炼层（未来）: 用户反馈 → 调整文本精炼策略

存储层内部:
  球体 ←→ FAISS（双向同步）
  球体 ←→ 层级（双向更新）
  球体 ←→ 角色表（双向引用）
  FAISS ←→ 角色表（检索时联动展开）
  FAISS ←→ 层级（检索时联动展开）
```

## 四、信息流的数据格式

每一层的接口格式决定了层间耦合的代价：

```
Layer 0 → Layer 1:  {source, raw_text, meta}
Layer 1 → Layer 2:  [{sentences, pronoun_marks, entity_candidates}]
Layer 2 → Layer 3:  [{fragment_id, text, explicit_subject, explicit_object, relations, quality_score}]
Layer 3 → Layer 4:  [{concept_id, embedding, level, parent_id, child_ids, relations}]
Layer 4 → Layer 5:  FAISS 索引 + Hierarchy 树 + RoleTable 表
Layer 5 → Layer 6:  [{concept_id, enriched_text, score, source_type, roles}]
Layer 6 → Layer 7:  [{text, subject, object, relations, score, source}]
Layer 7 → 用户:    {answer, sources: [{text, link}]}
```

## 五、反馈环 — 系统如何自我改进

```
用户交互产生的隐式反馈:

场景 A: 用户复制/收藏了某个答案
  → 生成层通知检索层: "这个答案的候选列表是好的"
  → 检索层提高对应候选的得分权重
  
场景 B: 用户追问「不对」
  → 生成层通知检索层: "这个答案的候选列表可能有误"
  → 检索层标记对应的候选为「待验证」

场景 C: 用户长时间在同一话题下检索
  → 检索层通知精炼层: "这个话题的用户关注度高"
  → 精炼层提高该话题下片段的质量评分（更多候选、更激进的主语注入）

场景 D: 无人问津的概念
  → 检索层通知存储层: "这个球体长时间未被检索到"
  → 存储层将该球体标记为「休眠」→ 降低在检索时的权重
```

## 六、质量瓶颈分析

```
全局瓶颈优先级:

P0（最高影响）: Layer 1–2 精炼层
  理由: 结构化片段的质量决定后续所有的角色表、层级、检索、生成
  当前状态: 规则 + 0.5B 选择题，准确率估计 70~80%
  瓶颈原因: 规则引擎的候选提取覆盖不全，0.5B 的选择有时偏向表层主语

P1: Layer 3–4 FAISS 向量质量
  理由: 语义检索的 recall 决定「能不能找到」相关概念
  当前状态: BGE-M3 embedding，质量稳定
  瓶颈原因: 无（BGE-M3 在中文上已经是当前最佳之一的本地模型）

P2: Layer 5–6 多样性排序
  理由: MMR 的 λ 参数敏感，场域检测的信号维度有限
  当前状态: v0.4 已经实现，效果可接受
  瓶颈原因: 需要更多用户反馈数据来调参

P3: Layer 7 生成层
  理由: 生成质量取决于前六层的输出
  当前状态: 多后端（Ollama / DeepSeek / Agent）
  瓶颈原因: 前六层质量上去后，生成层自然提升
```

## 七、Phase 路线图

```
Phase 1（当前 → v0.6）:
  Layer 0~2 全部实现（文本清洗 → 候选提取 → 片段注入）
  精炼层与存储层对接
  → 文本预处理质量成为可度量的指标

Phase 2（v0.7）:
  反馈环 Phase 1 实现（隐式反馈采集）
  检索层根据反馈调整排序
  → 系统开始「自我校准」

Phase 3（v0.8）:
  反馈环 Phase 2 实现（精炼层根据反馈调整策略）
  存储层对长时间不活跃的球体做衰减
  → 系统具备「遗忘能力」

Phase 4（v0.9 → v1.0）:
  完整的 Web UI（目前已有基础版）
  离线数据统计面板
  → 系统可独立运行，对非技术用户可用
```
