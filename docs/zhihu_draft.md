# 我给知识库装了一台「引力波探测器」— 从推理实验到生产代码的 3 小时

## 缘起：一个让我失眠的问题

三个月前，我搭了一个「重力知识库」——把 3000 多个知识球体挂在一个连通的引力场里，想实现「查一知三」的联想检索。

结果怎么都跑不通。球体之间几乎不连接，平均每个球体只有 1.7 个朋友。

我以为是阈值不对。调过 0.8 → 0.5，连接数从 24 涨到 39。但谁都看得出来，这不叫「联想」，这叫「勉强能找到邻居」。

直到一个朋友（一个 LLM，姑且叫它 Q）跟我说了一句话：

**「你的知识库缺的不是语义密度，是因果链。」**

这句话我消化了三天。

---

## 第一幕：推翻 Q 的假设

Q 提出了一套很漂亮的理论——因果链应该用「衰减临界角」来度量，阈值定在 60° 左右，跨句超过 3 个的才算「强因果」。

我说：漂亮归漂亮，跑个实验再说。

我建了一个「重力观测舱」——不要向量数据库，不要 GPU，只要 80 篇论文 + 一个轻量级嵌入模型。

流程极其简单：
1. 逐句扫描 80 篇论文，匹配因果标记词（propose / demonstrate / leads to / results in）
2. 找到因句后，在后续 2 句内找果句 → 得到 63 个因果链对
3. 在同一篇论文里随机抽取 102 个无关句对作为对照
4. 计算两两的余弦距离，换算成「衰减角」（角度越大 = 语义越远）

结果出来，我自己都愣了一下。

**因果链的平均衰减角是 54.24°，随机句对是 61.13°。**

两者相差 6.9°，p < 0.0001。不是巧合——因果链在语义空间里确实有特殊的结构。

但 Q 的 60° 阈值错了。只有 21% 的因果链 > 60°，绝大多数聚集在 45°-55° 之间。

更有趣的是：**所有因果链都在同一个段落内，平均只跨 2 个句子。**

没有一个跨段落、没有一个跨章节。

这个发现像一个锤子，砸在我一直以来最大的困惑上——为什么我的知识库不连接？因为我把整篇论文揉成一个球体，就像一个垃圾压缩机把所有的因果链压成了一个铁疙瘩。

**你要用显微镜看的东西，被你的仓库大门拦住了。**

## 第二幕：三个小时，从实验到代码

实验是两点开始的。推到 GitHub 的时候是四点半。

三个小时，我们写了六处代码改动：

### 1. 轴突连接检测（connections.py）

核心逻辑极其简单：
- 对每篇论文的每个段落，逐句扫描因果标记词
- 找到因句后，在同源文件的其他球体中匹配果句
- 匹配成功的建一条单向轴突连接，权重 0.35-0.6

12 条轴突连接，占全部连接的 31%。

### 2. 方向校准（connections.py + activation.py）

轴突连接必须是单向的——「吸烟」导向「肺癌」是对的，「肺癌」反向传导到「吸烟」是错的。

我用了一个纯规则引擎：
- 因句含 "leads to" / "enables" → 正向
- 果句含 "due to" / "attributed to" → 反向
- 无法判断 → 保守的双向

92% 的轴突被判断为正向。0% 反向。这符合直觉——学术论文几乎全是「提出方法 → 验证结果」的叙事结构。

在激活传播里，正向轴突不衰减（因果链完整传导），反向轴突半衰减（防止噪声扩散）。

### 3. 因果密度切分（chunker.py）

如果因果链只能在段落内存在，那切分的粒度就必须是段落级、因果链级。

新的切分策略：
1. 扫描每一段，问「这段有没有因果链？」
2. 有因果链的段落 → 独立成球体
3. 没有因果链的段落 → 合并到邻近的因果球体

一篇标准论文（Methods + Results + Conclusion + 背景段）被切成 2-3 个因果球体。

### 4. 修复了一个 bug

上传接口从未把 source_type 传递给切分器。意味着过去三个月导入的所有知识，走的都是默认的递归切分，从未触发过因果密度模式。

一行代码：
```python
raw_chunks = chunk_text(result.text, source_type=source_type)
```

三个月的 bug，一行修好。

## 第三幕：现在知识库长什么样

当前架构图（简化版）：

```
用户输入 Query
    ↓
[FAISS 语义检索] → 找到种子球体
    ↓
[激活传播] → 沿树突连接扩散 + 沿轴突连接因果传导
    ├─ 树突（联想）：dendrite，常规衰减
    └─ 轴突（因果）：axon，forward 不衰减 / reverse 减半
    ↓
[五层排序] → 质量 + 多样性 + 场域 + 冗余惩罚 + 连接惩罚
    ↓
[答案生成] → 检索上下文 + 大模型生成回答
```

球体间连接有六种策略：

| 策略 | 类型 | 权重 |
|------|------|------|
| 同簇语义 Top-3 | 树突 | 0.6 |
| 实体重叠 | 树突 | 0.4 |
| 跨簇嵌入相似 | 树突 | 0.3 |
| 时序相邻 | 树突 | 0.25 |
| 因果链（轴突） | 轴突 | 0.35-0.6 |
| 带方向 | 轴突 | forward/reverse |

当前规模：94 球体（含旧导入），68 球体（新实验数据）。

## 说实话，什么没做对

这篇文章避不开几个事实：

**1. 长论文测试没跑通**

因果密度切分需要长论文。我的 80 篇测试论文太短（600-800 字符），全被合并到一个块里。需要 3000+ 字符的论文才能看到多球体分裂的效果。

原因：arXiv API 不稳定（502 Bad Gateway），我没能拿到真实的多人论文。临时生成的测试论文篇幅不够。

**2. 连接仍然稀疏**

68 个球体，0 条连接。不是因为代码不对——是因为跨 16 个领域的短论文，两两余弦相似度不到 0.5。这是数据问题，不是架构问题。

**3. 三个小时太短了**

从实验设计到生产代码落地，三个小时能写出这些，但数据基建（论文获取、长文本测试）需要更长的时间投入。

## 现阶段不急着给它贴标签

说实话，这篇文章写到一半的时候，我犹豫了很久要不要用「世界模型」这个词。

最终决定不用的原因很简单：我拿 80 篇论文做了一个实验，证明了一个 6.9° 的语义差距，修改了一个阈值从 60° 到 55°，加了 12 条带方向的轴突连接——这些用数据说话就够了。不用给自己没做过的事情命名。

Q（那个提供外部分析的模型）说这个方向有潜力。我相信他的判断，但相信自己相信数据。

所以这篇文章到此为止，不讲愿景，只讲干了什么、发现了什么、改了什么代码。

以后真做出来了再说。
