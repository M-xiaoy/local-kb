# 大模型长文本复读机：不是 decoding 问题，是注意力坍缩

> 模型 SFT 后长文本生成频繁复读大段内容，repeat_penalty 调了没用，换 decoding 策略也没用，甚至在备选答案改写的模型里也会出现——这是为什么？

先说结论：**复读机问题不是 token 级别的决策错误，是注意力系统在长文本尾部发生了结构化退化。**

所有在 token level 做的修正——penalty、候选重选、温度调节——都治不了它，因为工具的输入已经坏了。

---

## 这个问题是怎么回事

想象你在一个很暗的房间里找东西。一开始灯光充足，你能看到整个房间。但随着时间推移，光源越来越弱，你能看清的范围越来越小。到最后，你只能看见脚下这一小块地板。你不敢往远处走，只能在原地打转。

模型长文本生成后期也是同样的情况。它的「注意力光源」在慢慢熄灭。

这不是一个原因造成的，是三个物理限制在同一点汇合。

---

## 第一个原因：BOS 吸走了大部分注意力

Attention Sink 是一个已经被系统验证过的现象，来自 Xiao et al.（2023）的工作。

Transformer 的注意力机制有个奇怪的统计偏差：**无论生成到多长，第一个 token（BOS）总是占据不成比例的注意力分数。**

原因很微妙。softmax attention 的分数之和必须为 1。到了长文本尾部，模型发现大部分内容和当前要生成的东西不相关。但它不能把所有注意力都集中在最近几个 token 上——softmax 会指数级放大高匹配分数。有一个 token 永远不算「完全不相关」，那就是 BOS。

```
正常生成时：
  attention 平均分布在约 100 个 token 上

到了 8K tokens 时：
  BOS → 40~55% 的注意力
  最近 50~200 个 token → 35~50%
  中间所有内容 → 剩下 10% 大家分
```

**模型实际上在用 35% 的注意力窗口做生成，剩下 55% 浪费在 BOS 上。** 那 10% 里恰好包含之前写过的内容——注意力顺着过去的路径走了回去，复读开始。

---

## 第二个原因：RoPE 的尺子不够长

现在主流模型（LLaMA、Qwen、DeepSeek 全系）都用旋转位置编码。RoPE 把位置编码成旋转角度——两个 token 的距离越远，它们的角度差越大。

模型训练时见过的最大角度差是有限的。到长文本尾部，角度差超出了它见过的范围：

```
训练时见过的角度差：不超过一个圆周
长文本尾部产生的角度差：可能到三个圆周
```

角度差太大，query 和 key 的点积会退化到接近零。**远处的 token 不是被模型「忽略」了，是模型根本没有能力给它们分配注意力。**

这就是为什么「改尺子」（YaRN / NTK-aware scaling）有效——它把位置编码压缩，让模型以为自己还在短距离区间里。代价是位置分辨率会稍微降低，但解决了视野缩小的问题。

---

## 第三个原因：脑容量满了

Transformer 的每一层都在叠加信息。到了长文本尾部，第 32 层的残差流里已经塞了 32 层 × 8K tokens 的信息。

LayerNorm 和激活函数对输入范围有硬限制。新信息进来，必须挤掉旧信息才能写入。模型的选择是：**挤掉中间的，保留两头的**——最近的内容和 BOS。

这就和 Attention Sink 形成了正反馈：BOS 本来就在吸注意力，中间的信息又被物理上挤出去了，注意力更集中到两端。两端越集中，越难挣脱。

---

## 三个机制互相加强

```
RoPE 尺子不够长 → 远处的看不见 → 注意力集中到近处
BOS 在吸注意力 → 近处再分走一半
脑容量已满 → 中间的记不住了

三者合在一起：
每生成一步，视野就缩小一点
缩到一定程度，就走不出来了
```

这不是三个独立的问题，是一个三角循环。任何一个在转，其他两个就会被带动。

---

## 那我现在怎么解决它

分两类：**不动训练的**（推理时加）和**动训练的**（改模型本身）。

### 不改训练的方案

**改尺子不改布料（YaRN / NTK-aware scaling）**

推理时改一个参数。模型以为自己还在短距离区间内，注意力系统不会触发「超出训练范围」的警戒。成本约等于零，几乎所有宣称长上下文的模型都在用这个。

可以理解为：给路灯加个放大镜，让光看起来还是亮的。

**不记住所有事（StreamingLLM）**

KV cache 只保留开头几个锚点 + 最近一段窗口，中间的扔掉。注意力窗口永远固定大小，不会膨胀。

代价是你忘掉了中间的内容——但聊天场景里你能记住半小时前说的每句话吗？记不住。StreamingLLM 模拟的就是这种自然遗忘。

**装一个复读检测器（LoopGuard, 2026）**

这是最新的思路，比 StreamingLLM 聪明。不提前截断 KV cache，而是实时监控注意力模式。当检测到「某个注意力头开始锁死在一段重复内容上」，才主动修剪重复部分的 KV cache。

LoopGuard 的作者们发现了一个微妙的问题：现有的 KV cache 淘汰策略依赖注意力分数来判断重要性——但模型开始复读时，重复内容的注意力分数反而很高，导致 cache 管理器**主动保留了会让问题恶化的内容**。

LoopGuard 打断了这个恶性循环。在测试集上，复读率降低了 90% 以上。

**让浅层监督深层（DoLa）**

Transformer 的不同深度在做不同的事。浅层做局部预测（这个词后面大概率接什么），深层做全局预测（上下文告诉我该写什么）。

正常生成时，浅层和深层的预测不一致。当模型开始复读时，它们变得高度一致——因为深层也视野缩了，看到的范围和浅层差不多。

DoLa 检测这个差异。差异小说明模型进入了迷茫区，主动抬高非复读候选的分数。代价是增加了一倍的推理开销，但效果显著。

### 改训练的方案

**找到复读的第一个字，改它（FTPO / Antidoom, Liquid AI, 2026）**

这是 Liquid AI 在 2026 年 7 月刚提出的方法。

复读不是随机发生的。统计发现，复读几乎总是从特定的词开始——在它们的 2.6B 模型上，最常触发复读的词是：

```
"the"    — 11.4% 的复读以此开始
"So"     — 4.5%
"Alternatively" — 3.2%
"Wait"   — 2.6%
"But"    — 2.5%
```

当模型不确定时，这些词是「安全的续写方向」——但进入这个方向后就走不出来了。

FTPO 的做法很直接：找到那个触发了复读的 token 位置，训练模型在那个位置更倾向于选择不会进入循环的续写方向。其他位置的偏好不动。

效果：复读率从 10.2% 降到 1.4%。而且评测分数全面上升——不是「牺牲质量换不重复」，而是「不重复本身就是质量」。

**用强化学习教模型写长文（LongWriter-Zero, ICLR 2026 Oral）**

清华大学的工作。核心思路是不在推理时修，而是在训练时用强化学习让模型学会长文本的结构。

入选 ICLR 2026 Oral，说明审稿人认可了这个方向的价值。

## 不同的视角：三级记忆体系

如果把三个原因映射到人的认知上，可以这样理解：

**第 1 级（显意识）** — 当前在想什么。对应因果注意力 + FFN，就是 Transformer 本身。所有复读发生在这里。

**第 2 级（近端记忆）** — 刚才的事。对应滑动窗口 / StreamingLLM。保持当前视野不膨胀。

**第 3 级（潜意识）** — 需要时才能想起来的东西。对应外部记忆检索。不在生成空间里存，需要时才拉取。

注意力坍缩只会在第 1 级发生。但如果第 2 级和第 3 级提供干净的信息，坍缩的信号无法形成正反馈——每次生成，模型都能从外部拉到一个和当前视野无关的新方向。

这个想法过于简化（实际的记忆检索和注意力交互比这复杂得多），但它指向了一个本质问题：**不是让注意力看得更远，而是让注意力不需要看那么远。**

---

## 总结

- 复读不是解码错误，是注意力系统在长文本尾部的结构化退化
- 三个原因（BOS 吸走注意力 / RoPE 尺子不够长 / 残差流塞满）互相加强
- 备选答案解决不了，因为候选全从退化后的视野采样，只是措辞不同
- 当前最佳实践：不改训练就叠 YaRN + LoopGuard + DoLa，改训练就走 FTPO 或 LongWriter-Zero
- 更长远的思路是记忆增强——不让注意力看它不需要看的东西
