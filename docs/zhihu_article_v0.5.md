# 重力知识库 v0.5：当知识库学会自己「长出」层级

## 四个问题，一个答案

上个月，我在迭代自己的重力知识库——一个以球体模型、场域引力、激活传播为核心的设计，不依赖现成 RAG 框架。迭代过程中冒出来四个问题，在某个深夜和小刘聊了一轮之后，我发现它们的根其实是一个：

**1. 语义检索能检测断裂，但无法重建逻辑**

你的 chunk A 讲「冰川融化导致海平面上升」，chunk C 讲「海平面上升威胁沿海城市」。embedding 余弦相似度能告诉你 A 和 C 都跟海洋有关——但无法知道 A 是因、C 是果。

**2. 切片越长，因果链越容易断在中间**

递归字符切片按分隔符优先级切，在一个 3000 字的段落里找不到 `\n\n` 就切到 `\n`，找不到 `\n` 就切到句号。因果链跨两段必断，断了就没法修。

**3. K-means 只能聚类，分不清含义维度**

「苹果」的科技侧和水果侧在向量空间中可能只差 0.2 的余弦距离。K-means 要么把它俩挤到一簇，要么随机漂到其中一簇——它没有语义区分能力。

**4. 来源多样性惩罚误伤高密度信息**

同一个文档切出来的 8 个 chunk 可能都命中 query，但多样性排序器按来源降权，第三个同源 chunk 就开始挨打。好的信息因为「同源」被砍了。

## 转机：把主语/宾语角色变成结构信号

传统 RAG 把所有文本切片同等对待，用 embedding 相似度衡量一切。我们的想法是：**句子中的实体在扮演不同语法角色时，本身就携带了结构信息。**

- 句子 A：「**冰川融化**（主语）导致**海平面上升**（宾语）」
- 句子 C：「**海平面上升**（主语）威胁**沿海城市**（宾语）」

同一个实体「海平面上升」在 A 中做宾语、在 C 中做主语——这不是巧合，这是推理链。A 和 C 通过这个实体构成了一条显式的因果路径，不需要 embedding 来猜。

我们把这个机制实现为 **角色共现表（RoleTable）**——记录每个实体在哪些句子中做主语、哪些做宾语。检索时，命中句子 A → 查它的宾语「海平面上升」→ 找到另一个以「海平面上升」做主语的句子 C → 加入到候选池。走的是**结构桥接**，不是余弦相似。

它解决的：

| 问题 | 之前 | 现在 |
|------|------|------|
| A 和 C 的因果链 | cosine < 0.3，查不到 | 实体角色桥接，不需要文本相似 |
| 跨文档因果 | 同文档内都难，跨文档基本不可能 | 实体跨文档出现就能桥接 |
| 切片断裂 | 断了就断了 | 断在 A 和 C 的因果链通过实体重建 |

## 更进一步：知识库自己长出层级

角色桥接做到后面，我们想：能不能让知识库自己判断哪些概念值得升格为「一级球体」，哪些应该留在「二级」？

**做法不是规则，是社区检测。**

所有句子（二级球体）构成一张图，边来自：
- **实体共现边**：两个句子包含同一个实体
- **向量相似边**：embedding 余弦 > 0.65

然后跑 **Label Propagation**（标签传播）社区检测——每个节点的标签向其邻居扩散，直到收敛。结果是若干社区，每个社区内的话语绕着相同的实体转。

```
社区 A（21个句子）：气候变化的证据 → 高频主语：「气候变化」「冰川融化」
社区 B（15个句子）：能源转型的方案 → 高频主语：「可再生能源」「碳中和」
社区 C（3个句子）：孤立的一句话 → 不划等级
```

**对每个社区独立判断**，不是全局一刀切：

```
if 社区大小 < 5 → 不划等级（太小，没有统计学意义）

覆盖阈值 = max(2, 社区大小 × 0.33)
if 主语出现次数 < 覆盖阈值 → 不够主流，不升格
if 一级球体已 ≥ 3 个 → 够了，后面的降级

社区300句 → 主语需出现≥99次 → 真正的大主题
社区10句  → 主语需出现≥4次 → 小社区适度放宽
```

这样 **300 句的大社区不会冒出 50 个一级球体**，而 **10 句的小社区也能抓到核心概念**。

一级球体本身也是一个向量点（隶属社区的语义质心），和二级球体共享同一 FAISS 索引。检索时命中一级 → 展开到它的二级子球体；命中二级 → 可上卷到一级。

## 做个对比

| 维度 | 经典方案 | 我们 |
|------|---------|------|
| 因果链 | embedding cosine（猜） | 实体角色桥接（可验证） |
| 层级 | 预设（目录树/LSH） | 社区检测自然生长 |
| 阈值 | 全局固定 | 每社区独立计算 |
| 主语抽取 | spaCy/哈工大 LTP | 规则引擎（够用即可） |
| 一级球体数 | 不想清楚不动 | 最多3个/社区，控制质量 |

## 一点体会

这套方案不是突然想出来的。它是从「为什么 K-means 聚类不好用」一步步推到「社区检测 + 角色建模」的。过程中的每一步都基于一个简单原则：**不做结构做不到的事；能做结构的绝不让模型猜。**

社区检测这个技术本身不新（Label Propagation 是 2007 年的论文），但它在这种场景下的应用——**用角色分布 + 向量相似度的混合图做社区检测，再把社区的语义中心提炼为概念球体**——似乎没有现成的做法。

目前这套已经在我们的知识库（4452 个球体，10846 条实体关系）上跑通，启动时 ~15 秒完成全量生长。代码已开源：[M-xiaoy/local-kb](https://github.com/M-xiaoy/local-kb)，有兴趣的可以看 `pipeline/role_table.py` 和 `pipeline/hierarchy.py`。

下一阶段的方向：增量社区更新（不是每次全量重跑）、一级球体之间的语义关系（不同社区的概念可能共享子球体）、以及让角色表的跳转权重根据用户反馈动态调整。
