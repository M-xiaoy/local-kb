# 重力知识库·架构 v2 — 球体空间视角

> 不是 RAG 流水线，是知识球体的引力场。
> 每一层都是主流技术在球体空间中的「翻译」。

---

## 一、世界观：一套隐喻解释所有

```
主流 RAG 的技术堆栈 → 重力球体的自然现象

文本片段         → 球体 (Sphere)
语义相似度       → 引力 (Gravity)
层次关系         → 轨道层级 (Orbit Level)
共现关系         → 社交网络 (Social Graph)
检索             → 信号传播 (Signal Propagation)
多样性与冗余关系  → 质量场叠加 (Field Superposition)
用户个性化定制    → 观测者效应 (Observer Effect)
知识积累和遗忘    → 引力衰减与球体风化
```

**这不仅仅是比喻——所有组件都按这套隐喻设计和命名，代码、接口、注释、配置项都使用同一套语言。**

## 二、球体空间的核心设施

### 2.1 球体（Sphere）

```
球体 = 知识的最小语义单元。

字段:
  concept_id: "gravity_knowledge_base"     ← 唯一标识
  name: "重力知识库"                        ← 展示名
  description: "基于重力空间检索的RAG系统"    ← 语义描述
  embedding: [0.23, -0.45, ...]            ← 位置矢量（1024维）
  level: 1                                  ← 轨道层级（0=无层级, 1=一级, 2=二级）
  parent_id: "rag_system"                   ← 围绕谁旋转
  child_ids: ["sphere_store", "faiss_store"] ← 谁围绕它旋转
  connections: [                            ← 社交网络（连接其他球体）
    {"target": "retrieval_pipeline", "strength": 0.85, "type": "共现"},
    {"target": "embedding_model", "strength": 0.60, "type": "依赖"}
  ]
  term_weights: {"重力": 2.5, "知识库": 2.0, "检索": 1.8}  ← 术语引力
  cluster_id: 3                              ← 星系归属
  source_type: "technical_note"             ← 来源类型
  quality_score: 0.85                       ← 质量评级
  last_accessed: "2026-07-16"               ← 最近被观测时间
```

### 2.2 力（Force = 检索信号）

```
检索信号不是「关键词匹配」，是力。

引力的来源（检索触发信号）:
  ├── 查询文本的 embedding 矢量（主引力源）
  ├── 查询关键词的 term_weights（辅助引力场）
  ├── 用户设定的场域权重（观测者干预）
  └── 激活传播的接力信号（社交网络扩散）

引力的计算（球体受到的合力）:
  F_total = α × F_semantic + β × F_term + γ × F_field + δ × F_social
  
  F_semantic = cosine(query_embedding, sphere_embedding)
  F_term     = match(query_keywords, sphere.term_weights)
  F_field    = user_field_weight × match(field, sphere.source_type)
  F_social   = sum of activated neighbor forces (已激活球体传过来的力)
  
  α, β, γ, δ = 可配置权重，默认为 0.7, 0.3, 0.0, 0.0
```

### 2.3 轨道（Orbit = 层级结构）

```
球体的层级不是人为指定的，是在共现数据中自然「生长」出来的。

Label Propagation 社区检测:
  在共现图中检测社区 → 每个社区是一组密集连接的球体
  
社区内部分层:
  覆盖度 > 33% 的球体 → 一级（中心天体，在引力场中占主导）
  覆盖度 ≤ 33% 的球体 → 配置的子级（围绕一级旋转）
  上限: 每个社区最多 3 个一级

轨道展开（检索时）:
  命中一级球体 → 自动展开其二级子球体（就像打开一个星系看里面的行星）
```

### 2.4 场（Field = 来源分类 + 用户偏好）

```
每个球体有来源类型（technical_note / paper / conversation / web）

用户可以对特定来源类型设权重:
  field_weights = {"technical_note": 1.0, "paper": 1.2, "conversation": 0.5}
  
  检索时: field_score = field_weights.get(sphere.source_type, 1.0)
  最终分数: F_total = 0.7 × F_semantic + 0.3 × (F_term × field_score)
```

## 三、主流技术在重力空间中的翻译

| 主流技术 | 在重力空间的翻译 | 差异 |
|---------|----------------|------|
| **Hybrid Search** (BM25 + 稠密) | 不是两路并列检索。BM25 信号融进 `term_weights`，在 `keywords.py` 的术语引力匹配中使用 | 词汇匹配作为连接权重的一部分，不是独立检索通道 |
| **Cross-encoder Reranker** | 可选模块 `reranker.py`，Ollama 本地评分 1~5 | 不依赖 GPU，轻量化部署，配置可关 |
| **RAPTOR** (层次树) | k-means 簇质心 = 自然的高层节点。簇摘要未来可 LLM 生成 | 质心由聚类自动产生，不是递归生成的摘要树 |
| **GraphRAG** (微软) | `connections` 表 + `entities` 标注 = 轻量图。`bridge.py` 做双向 BFS 路径发现 | 图在球体层不在实体层，更细粒度 |
| **Self-RAG** | 不训练反射 tokens。用 `gravity_focus` 做隐式置信度——检索结果场域分散 → 不可信 → 可触发二次检索 | 靠空间拓扑做置信度，不需要微调 |
| **HyDE** | `rewriter.py` 的 query rewrite 模式之一：生成假设答案 → 用假设答案检索 | 仅在 deep 模式启用，Ollama 本地执行 |
| **CRAG** | 置信度评分机制尚未实现 | v2 roadmap Phase 2 |

## 四、独特价值（主流没有的）

### 4.1 球体社交网络

```
不是实体图（实体和实体之间有关系），
是知识片段之间的关系网。

常规知识图谱:
  节点 = 实体（"张三"、"苹果公司"、"深度学习"）
  边 = 关系（"任职于"、"生产"、"研究"）

球体社交网络:
  节点 = 语义片段（"重力知识库的检索质量很高"）
  边 = 共现关系（两个片段在文档中相邻出现）
  + 依赖关系（片段 A 引用了片段 B 的概念）
  + 层次关系（片段 A 是片段 B 的父概念）

优势:
  - 不需要预定义 schema（不需要先定义实体类型和关系类型）
  - 片段级的连接比实体级更细粒度
  - 支持「本来没有直接关系，但因为共现而被连接」
```

### 4.2 激活传播

```
检索从单点匹配变成信号扩散。

常规检索:
  query → 找相似的球体 → 返回

激活传播:
  query → 找相似的球体 → 排序
       ↓
  从排序靠前的球体出发 → 沿 connections 传播
       ↓
  传播到的球体参与排序（加传播系数衰减）
       ↓
  重复传播（最多 3 跳）

效果:
  "检索质量" → 找到"重力知识库" → 扩散到它的连接
  → "FAISS索引"、"embedding模型"、"多样性排序"
  → 发现原本不在直接匹配范围内的隐式关联
```

### 4.3 五层排序

```
不是一次排序定生死，是五层过滤。

第 1 层: FAISS 语义相似度（召回层，取 top-50）
第 2 层: 角色展开 + 层级展开（扩展层，取 top-50→top-80）
第 3 层: 场域加权 + 术语引力（精排层，重算得分）
第 4 层: MMR 多样性排序（去冗余层，合并 top-K）
第 5 层: 连接度 + 活跃度（终排层，活跃球体优先）

每层有独立的可配置参数:
  top_k, lambda, field_weights, decay_factor
```

### 4.4 内部路由工具

```
知识库不只是检索器，还是可探索的空间。

检索器模式:
  输入 query → 返回排名靠前的片段
  用途: 问答、文档检索

探索器模式:
  输入 query → 返回相关球体的社交网络（图结构）
  用途: 发现概念之间的关系、知识图谱可视化

巡游器模式:
  无输入 → 从随机球体出发，沿 connections 游走
  用途: 随机漫步发现新联系（类似 Obsidian 的 graph view）
```

## 五、精炼层在重力空间中的位置

回到刚才讨论的文本预处理——它在重力空间中的角色是**球体的铸造车间**。

```
原始文本颗粒
    ↓
Layer 0: 清洗 → 去掉杂质（代码块/格式标记/噪声）
Layer 1: 分析 → 标记结构（句子边界/代词位置/引号内容）
Layer 2: 候选提纯 → 确定每个主语和宾语在空间中的身份
Layer 3: 注入装配 → 主语和宾语作为球体的两个锚定面

关键:
  精炼层产出的不是「文本」，是球体的雏形——带anchor的球形胚体
  主语 = 这个球体绕谁转（父球体候选）
  宾语 = 这个球体和谁有连接（社交连接候选）
  
  精炼层的质量 ≈ 球体铸造的精度
  精度不够 → 球体位置偏差 → 引力场紊乱 → 检索信号失真
```

## 六、路线图（重力空间语言）

```
Phase 1 — 铸模（当前 ~ v0.6）
  精炼层 vs 存储层
  目标: 球体铸造精度 > 90%

Phase 2 — 校准（v0.7）
  反馈环
  目标: 引力场自校准（查询历史 → 调整 αβγδ）

Phase 3 — 风化（v0.8）
  遗忘机制
  目标: 球体长期不活跃 → 引力衰减

Phase 4 — 星图（v0.9 → v1.0）
  探索器 + 巡游器
  目标: 知识库可作为可探索空间使用
```
